# Настройка FFmpeg, seeders и публикация проекта

## Цель

Настроить только приложение LUT Web на использование FFmpeg 6.1 через переменные окружения, привести начальное наполнение базы к воспроизводимому состоянию, проверить реализацию относительно проектных требований и безопасно опубликовать содержательные изменения в ветку `main`.

## Конфигурация FFmpeg

- Рабочий бинарник: `/www/server/ffmpeg/ffmpeg-6.1/ffmpeg`.
- Команда `ffmpeg6` уже указывает на этот бинарник, но приложение должно использовать абсолютный путь, чтобы queue workers и systemd-сервисы не зависели от `PATH`.
- `LUT_TESTER_FFMPEG_BINARY` и `CUSTOM_LUT_FFMPEG_BINARY` задаются в production `.env`.
- Эти же ключи документируются в `.env.example` и `deploy/.env.production.example`.
- Storefront media использует общий бинарник из `lut-tester.ffmpeg_binary`; отдельная переменная для него не создаётся.
- После изменения выполняются `config:cache`, перезапуск очередей и прикладные doctor/self-test команды.

## Стратегия seeders

- Seeders создают воспроизводимые справочные и демонстрационные данные, необходимые для проверки основных пользовательских сценариев.
- Модели аудита, очередей, уведомлений, webhook-событий и платёжных транзакций не обязаны иметь отдельный production seeder: это операционные данные, возникающие в ходе работы приложения.
- Аудит сопоставляет каждую модель с production seeder, demo seeder, фабрикой либо обоснованно исключает её из начального наполнения.
- `DatabaseSeeder` остаётся безопасным production baseline. Демонстрационное наполнение подключается только если это уже предусмотрено конфигурацией проекта или требуется существующей спецификацией.
- Полный reseed выполняется командой `php artisan migrate:fresh --seed --force` только после резервной копии точного SQLite-файла.

## Git и файловые права

- Случайные изменения `100644 -> 100755`, появившиеся при распаковке ZIP, не публикуются и нормализуются по Git index без изменения содержимого.
- Перед коммитом отдельно проверяются все содержательные diff и новые файлы.
- Не публикуются `.env`, `lutweb.zip`, `.user.ini`, `.well-known`, кэши, логи и другие server/runtime-артефакты.
- Публикуются подтверждённые исходники, конфигурационные шаблоны, тесты, seeders и документация.

## Проверка

- Конфигурационные тесты подтверждают получение FFmpeg из env/config.
- Seeders проверяются сначала на изолированной тестовой SQLite-базе.
- Выполняются Pest, TypeScript conformance/type-check, frontend build, Laravel doctors и self-tests.
- После production reseed проверяются количества и связи основных записей, миграции, systemd workers, scheduler, HTTPS, health endpoint и Vite asset.
- Незавершённые требования ищутся в документации, тестах, doctor-командах и явных TODO/TBD; внешние интеграции без credentials фиксируются как эксплуатационные ограничения, а не реализуются фиктивными значениями.

## Безопасность и восстановление

- Перед `migrate:fresh` создаётся датированная копия `database/database.sqlite` вне публичного каталога.
- Production `.env` остаётся вне Git и с ограниченными правами доступа.
- В Git не попадают секреты, реальные платёжные данные или серверные файлы.
